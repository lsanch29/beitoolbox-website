Registries and Reproducibility

Registries are the toolbox mechanism for tracking pipeline state across GUI interactions. They provide a single source of truth for whether a dataset or study has been preprocessed, has energy artifacts, or has been tested with a model. Registries also support reproducibility by linking status to persistent outputs.

Role of registries

In a GUI-driven scientific workflow, state can become fragmented across callbacks, cached structs, and file system artifacts. Registries solve this by enforcing:

  • Centralization: one authoritative status store per workflow stage.
  • Auditability: status can be printed, inspected, and saved.
  • Deterministic enabling: menus are enabled based on registry state, not on ad hoc flags.

Registry types

The toolbox uses three primary registries stored in figure appdata:

BEI_preproc_reg

Tracks whether a dataset or study input has valid preprocessing outputs available.

BEI_energy_reg

Tracks whether a study has computed energy artifacts (PDFs, covariance, cached features).

BEI_testing_reg

Tracks whether a dataset or study has been evaluated with a selected model and where results were saved.

Registry manager

A small utility (for example bei_registry) mediates get, set, update, and remove operations.

Record schema

Each registry stores an array of records. A minimal record schema is: 

record.ObjectName   % string key, usually "DATASET|MODALITY|DIAG" or "STUDY|<NAME>"
record.Status       % string label, for example "Preprocessed", "EnergyReady", "Tested"
record.UpdatedAt    % timestamp (recommended)
record.ArtifactPath % path to PDO/PDI/energy/testing output (recommended)

The critical scientific property is key stability. The same object should map to the same registry key across sessions, so the GUI can correctly infer what work has already been completed.

Provenance

Registries should not replace provenance in output files. Instead, they should point to artifacts that carry full provenance. At minimum, output artifacts should store:

  • Toolbox version string
  • Timestamp
  • Configuration parameters relevant to the artifact
  • Resolved cohort membership identifiers (for PDI and testing results)
Avoid stale cached fields

Cached processingStatus fields inside dataset structs can drift from reality. Use registries as the enabling logic and treat cached fields as display-only.

Reproducible sessions

A reproducible analysis session is one where:

  • All required dependencies are detected (MATLAB toolboxes and external toolboxes such as EEGLAB, SPM12, FieldTrip).
  • Inputs are resolved through stable dataset and study definitions.
  • All derived artifacts are versioned and stored under predictable folders.
  • Registries correctly point to the latest artifacts and can be printed for audit logs.
