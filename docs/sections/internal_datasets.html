Internal Datasets

Internal datasets support a common population-analysis requirement: including a stable reference cohort (often healthy controls) that is not currently registered as a standard dataset in the GUI. This allows studies to combine externally loaded datasets with a persistent internal reference, without forcing the internal cohort to appear as a first-class dataset entry in every session.

Concept

An internal dataset is a pseudo-dataset representation constructed from subjects already present in a study PDI. The key rule is that internal subjects are those whose dataset field is not one of the currently loaded dataset names. Operationally, internal datasets are discovered by comparing:

  • Registered dataset names: what the GUI currently knows as top-level datasets.
  • PDI subject dataset tags: the dataset label stored in each patient_data entry.

Why internal datasets exist

Internal datasets improve rigor and usability when:

  • You maintain a curated healthy control library that should remain stable across studies.
  • Controls include multiple modalities and you want their subject counts visible in population plots and model testing.
  • You need reproducible population metrics without requiring every contributing dataset to be loaded in the current session.
Scientific motivation

Equal Energy Distribution tests and coupling proxies depend on reference distributions. A stable internal control cohort reduces variance introduced by shifting baselines across analyses.

Mechanics

In a typical workflow, a study loads its latest PDI and then constructs internal dataset entries on demand. These entries are used only for plotting and counting, not for recomputing subject PDOs. A practical convention is: 

Internal subject if: subject.dataset not in {datasets.name}

Internal datasets should preserve meaningful stratification. For example, if the internal cohort includes healthy controls recorded in multiple modalities, you should keep those strata distinct in reports (for example HC_EEG and HC_fMRI) if your downstream models or plots treat them differently.

Subject counts and reporting

Correct subject counts are a reproducibility requirement. The toolbox should report study membership using the resolved PDI, not the currently loaded dataset structs. In particular:

  • Population and subject-level plot menus should count internal subjects when building study summaries.
  • Model testing menus should match the same counting logic so that reported denominators are consistent.
Consistency rule

If two menus show different subject counts for the same study, at least one is not using the PDI as the source of truth. The study snapshot should be authoritative for all population-level reporting.

Common failure modes

  • Silent label drift: if internal controls are updated without writing a new PDI snapshot, the study can become non-reproducible.
  • Duplicate subject codes: the same subject appearing under multiple dataset tags can inflate counts and bias group statistics.
  • Missing required fields: internal entries lacking patient_code, diagnosis, or time series fields can break aggregation code.
  • Modality conflation: combining internal HC across modalities without stratification can blur biologically meaningful differences.
